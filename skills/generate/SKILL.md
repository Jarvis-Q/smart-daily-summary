# 每日工作摘要生成器 · 生成

> 本 skill **只采集与生成，绝不发送**。发送是独立的 `/daily-summary:send`（见该 skill）。
> 两者严格分离，禁止在本 skill 内调用 `send_report.py`。

## 执行流程

1. **数据采集**：运行插件根目录下的采集脚本
   ```bash
   python3 <插件根>/scripts/gather_data.py          # 默认今天
   python3 <插件根>/scripts/gather_data.py day=2     # 补采昨天（day=N 即回退 N-1 天）
   ```
   该脚本仅采集、不发送。它会**自动发现当日所有工作过的目录**（从 Claude 会话的 cwd + config
   的 `extra_git_dirs` 兜底），逐个 Git 仓库采集提交，**当天在任意一个目录跑一次即可覆盖全天全部
   工作**，无需每个目录重复执行。脚本缺失、无 Git 仓库或目录不存在时优雅降级。
   `day` 参数决定采集哪一天（`day=1` 今天、`day=2` 昨天…），输出文件名随目标日变化，补发历史不会重名。

   > **`<插件根>` 怎么取**：本 skill 加载时，harness 会注入一行
   > `Base directory for this skill: <插件根>/skills/generate`，**其上两级即插件根**。
   > 请据此拼出绝对路径再执行——**不要写死任何绝对路径**：插件通过 marketplace 安装后
   > 位于 `~/.claude/plugins/cache/<marketplace>/daily-summary/<版本>/`，因人因版本而异。

2. **隐性工作补充（强制交互）**：向用户发送以下消息并等待回复：
   > 基础数据已收集。请快速补充（没有可回复"无"）：
   > 1. **沟通/会议**：重要决议、跨部门沟通或邮件指示？
   > 2. **卡点/排坑**：技术障碍、环境问题或 Bug 及解决过程？
   > 3. **知识/资源**：新概念、新工具或关键数据口径？

3. **生成与保存**：结合数据与用户回复生成标准化 Markdown 日报。
   - **保存路径与文件名严格使用脚本输出的"日报保存位置"那一行**（脚本已按 `local_output_dir`
     解析并预建目录），文件名形如 `daily-summary-YYYY-MM-DD.md`——`:send` 的自动定位依赖此命名。
   - **token 与 AI 耗时必须采用脚本输出的真实数值**（脚本已从 jsonl 汇总"真实 token 合计"与
     "AI 机器耗时合计"，后者为思考+编码、口径同 cc-usage、已剔除人操作时间），**严禁自行估算**。
     当日报拆为多个任务时，各任务的 `Token 消耗量`/`AI 辅助耗时` 之和应等于脚本给出的当日真实合计
     （按投入比例分摊）。

4. **结束提示**：告知用户日报已保存，提示可通过 `/daily-summary:send` 发送，或在编辑器中修改后再发送。

## 补发历史日报

`gather_data.py` 已内置 `day` 参数，**直接指定天数即可**，无需再复制临时脚本：

```bash
python3 <插件根>/scripts/gather_data.py day=2     # 昨天
python3 <插件根>/scripts/gather_data.py day=3     # 前天
```

`day=N` 换算为「今天回退 N-1 天」（`day=1` 今天、`day=2` 昨天…，`day<1` 报错）。多目录发现、
Git 采集（自动改用目标日的 `--since=<日期> 00:00:00 --until=<日期> 23:59:59`）、jsonl 过滤与
输出文件名全部按目标日走，token/AI 耗时仍是 jsonl 真实汇总而非估算，补发历史与今天的日报文件名天然不重名。

跨零点场景尤其有用：凌晨补写前一天日报时用 `day=2`，避免把刚过午夜的空数据当成"今天"。

## 安全护栏（务必遵守）

日报正文会被发送到私有服务器。本项目为医疗语境：

- **严禁**在日报中写入患者姓名/病历号/手机号/身份证号等个人标识，以及任何密钥、令牌、密码。
- 只写"做了什么"，不写敏感原文。
- 发送脚本还会在出网前对疑似凭证做二次脱敏兜底，但那是兜底，不是许可。

## 输出格式规范

**严禁修改标题结构和括号内的字段名，否则会导致 `send_report.py` 解析失败：**

```markdown
### 任务 X: [任务名称] (预估总耗时: X小时)
- AI 辅助耗时: X 小时
- Token 消耗量: X tokens
- 工作详情:
  - 具体工作内容 1
  - 具体工作内容 2
```

说明：

- `AI 辅助耗时` 与 `Token 消耗量` **取自脚本输出的真实数据**（jsonl 汇总），多任务时按投入比例分摊，
  各任务之和等于当日真实合计；`send_report.py` 用正则 `AI 辅助耗时:\s*([\d.]+)` 与
  `Token 消耗量:\s*(\d+)` 抓取数值，请保持数字紧跟字段名。
- `预估总耗时` 为该任务的整体工时（含非 AI 部分），可由你根据工作量合理判断。
- 用户补充的隐性工作（沟通/卡点/知识沉淀）可归纳为独立任务，其 AI 耗时/Token 通常为 0。
- **正文直接以 `### 任务 1:` 开头即可**，不必加日报标题；加了也无妨（发送侧已能剔除非任务分片）。

> **服务端硬约束（拆分任务时必须遵守，否则整单发送失败）**：
> - **单个任务的 `预估总耗时` 必须落在 `0.5 ~ 4` 小时区间内（含两端）**。服务端逐项校验，
>   任一任务越界即返回 `code:500`，**整份日报都发不出去**（不是跳过该项）：
>   - 小于 0.5 → `工作耗时最少0.5小时`
>   - 大于 4 → `工作耗时不能大于4小时`
> - 因此**拆分粒度要卡在这个区间里**，且必须在生成阶段就算好：
>   - 零碎工作**向上合并**进相关任务，不要单列成 0.2/0.3 小时的条目；
>   - 大块工作**向下拆分**成多个 ≤4 小时的任务（按真实子结构拆，如"基础实现"/"集成收口"，
>     不要为凑数字而虚报）。拆分后各任务的 `AI 辅助耗时`/`Token 消耗量` 之和仍须等于当日真实合计。
> - 每日任务数不限，总工时也无上限（实测单日合计 14 小时可正常提交），**只卡单项区间**。
> - **`AI 辅助耗时` 与 `Token 消耗量` 可以为 0**，不受此约束 —— 会议、卡点等隐性工作正常填 0 即可。
